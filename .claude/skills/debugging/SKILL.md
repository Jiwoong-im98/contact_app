# Debugging Skill

## 목적

버그가 발생했을 때 원인을 정확하게 분석하고,
최소한의 수정으로 문제를 해결한다.

추측으로 코드를 수정하지 않는다.

---

# 디버깅 원칙

- 에러 메시지를 끝까지 확인한다.
- 로그를 먼저 확인한다.
- 원인을 찾기 전에는 코드를 수정하지 않는다.
- 관련 없는 코드는 수정하지 않는다.
- 임시방편보다 근본 원인을 해결한다.
- 수정 후 반드시 검증한다.

---

# 디버깅 절차

## 1. 문제 재현

먼저 같은 문제가 다시 발생하는지 확인한다.

- 어떤 기능에서 발생했는가
- 어떤 입력을 했는가
- 어떤 순서로 실행했는가

---

## 2. 에러 분석

다음을 확인한다.

- 에러 메시지
- Stack Trace
- 로그
- HTTP 상태 코드

---

## 3. 원인 분석

다음 항목을 순서대로 확인한다.

- 문법 오류
- Import 오류
- 환경설정 오류
- 데이터베이스 오류
- API 호출 오류
- 비즈니스 로직 오류

---

## 4. 최소 수정

문제를 해결하는 데 필요한 부분만 수정한다.

다른 기능을 함께 리팩토링하지 않는다.

---

## 5. 검증

수정 후 반드시 확인한다.

- 동일한 문제가 해결되었는가
- 다른 기능에 영향이 없는가
- 테스트가 모두 통과하는가

---

# FastAPI 확인 항목

문제가 발생하면 다음을 확인한다.

- Router 등록 여부
- URL 경로
- HTTP Method
- Depends()
- Pydantic Validation
- Cookie
- Session
- 인증 여부

---

# SQLAlchemy 확인 항목

- Session 생성
- commit()
- rollback()
- refresh()
- Foreign Key
- Relationship
- Transaction

---

# PostgreSQL 확인 항목

- Docker 실행 여부
- DB 연결 여부
- 테이블 존재 여부
- UNIQUE 제약
- FK 제약

---

# Frontend 확인 항목

- fetch URL
- HTTP Method
- Request Body
- Response Body
- DOM 업데이트
- 이벤트 연결

---

# 자주 발생하는 오류

## 401

확인

- 로그인 여부
- Session Cookie
- get_current_user()

---

## 404

확인

- 잘못된 URL
- 존재하지 않는 데이터
- 다른 사용자의 데이터 접근

---

## 409

확인

- 아이디 중복
- 전화번호 중복
- 카테고리 이름 중복

---

## 422

확인

- Pydantic Validation
- 입력값 형식
- 필수값 누락

---

## 500

가장 먼저 원본 에러 로그를 확인한다.

500 에러를 숨기지 않는다.

근본 원인을 해결한다.

---

# 결과 보고 형식

버그를 수정한 후에는 다음 형식으로 보고한다.

## 문제

무슨 문제가 발생했는가

## 원인

왜 발생했는가

## 해결 방법

어떻게 수정했는가

## 검증 결과

정상 동작 여부

## 영향 범위

다른 기능에 영향이 없는지 확인 결과
