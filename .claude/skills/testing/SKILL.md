# Testing Skill

## 목적

기능이 정상적으로 동작하는지 검증하고,
수정으로 인해 기존 기능이 깨지지 않았는지 확인한다.

모든 테스트는 재현 가능해야 하며,
테스트 결과는 누구나 동일하게 확인할 수 있어야 한다.

---

# 기본 원칙

- 구현한 기능은 반드시 테스트한다.
- 버그를 수정했다면 재발 방지 테스트를 추가한다.
- 테스트 없이 완료했다고 판단하지 않는다.
- 테스트는 독립적으로 실행 가능해야 한다.
- 테스트는 실제 사용자의 관점에서 작성한다.

---

# 테스트 우선순위

다음 순서로 테스트를 수행한다.

1. 단위 테스트 (Unit Test)
2. API 테스트
3. 통합 테스트 (Integration Test)
4. E2E 테스트 (Playwright)
5. 회귀 테스트 (Regression Test)

---

# 단위 테스트

확인 항목

- 함수
- Service
- Validation
- Utility

검증 내용

- 정상 입력
- 경계값
- 잘못된 입력
- 예외 처리

---

# API 테스트

모든 API는 다음을 확인한다.

- HTTP Method
- URL
- Request Body
- Response Body
- Status Code
- 예외 상황

---

# Playwright 테스트

사용자 관점에서 테스트한다.

예시

- 회원가입
- 로그인
- 연락처 등록
- 연락처 수정
- 연락처 삭제
- 카테고리 CRUD
- 로그아웃

버튼 클릭, 입력, 화면 변경까지 검증한다.

---

# 회귀 테스트

기능 수정 후 반드시 확인한다.

- 기존 기능이 정상 동작하는가
- 새로운 기능이 기존 기능에 영향을 주지 않는가

---

# 테스트 체크리스트

기능 완료 전 반드시 확인한다.

□ 정상 동작

□ 예외 처리

□ 입력 검증

□ 권한 확인

□ 데이터 저장

□ 화면 갱신

□ 에러 메시지

□ 기존 기능 영향 없음

---

# 테스트 결과 보고

다음 형식으로 결과를 작성한다.

## 테스트 대상

예) 연락처 등록

## 수행한 테스트

- 정상 등록
- 전화번호 중복
- 잘못된 전화번호
- 로그인 없이 접근

## 결과

- 성공
- 실패

## 발견된 문제

문제가 있다면 원인과 해결 방안을 함께 작성한다.
